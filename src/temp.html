Overall

이 페이지는 Windmill 서버 API 명세서로, TBroad 용으로 커스터마이즈 된 버전이다.

Chrome 웹 브라우저의 webstore에서 Postman REST client 를 사용한다면 개발자를 위해 만들어 놓은 collection을 import해서 사용할 수 있다.

접속

TBroad의 윈드밀 서버 API는 서버 종류에 따라 세 종류(UPMS, ACMS, Search)로 구분이 되어 있다.

기본 URL

기본 URL 형식

http://<host>[:<port>]/<server>/api1/<resource>

host는 windmill server의 호스트가 된다. TBroad에서는 192.168.10.10으로 지정되어 있다.
개발자용으로 사내에서 운용되는 서버는 현재 10.10.0.180이다(변할 수 있음)
port는 표준 HTTP 포트인 80 포트를 사용하므로 80으로 지정해도 되나 굳이 지정하지 않아도 default 80으로 지정된다.

서버 및 API 종류 구분

  • server : 각 서버 명칭과 서버가 제공하는 API 종류는 다음과 같다.
    • UPMS : userprofile
      • auth : 사용자 및 기기 인증
      • accounts : 사용자 계정 및 pairing
      • user-devices : 사용 기기
      • user-photos : 사진
      • user-mycontents : 즐겨찾기
      • user-purchase : 구매
      • user-watch : 시청
      • user-services : 서비스 정보
    • ACMS : acms
      • menu : 메뉴 구성
      • mails : 공지사항
      • category : 카테고리 정보
      • campaign : 켐페인
      • channel,schedule : 채널 및 스케쥴 정보
      • programs : 프로그램 정보
      • packages : 패키지 정보
      • apps : 어플리케이션 카탈로그
      • reviews : 컨텐트 리뷰
      • pictures : 포스터 이미지
    • Search : search
      • search : 키워드 검색
      • complete/search : 추천 검색어
      • keyword/popular : 실시간 검색어
      • search_history : 검색 히스토리

  • 상황: 메뉴 전체 구조를 가져올 때
  • 사용되는 API : GET menus
  • 필요한 input parameter : child=all (parameter가 없을 경우 1-depth만 가져옴)
  • access_token: 개발용(_tbroadme_)
  • 접속 호스트 : 사내 개발용 (10.10.0.180)
  • URL: http://10.10.0.180/acms/api1/menus?child=all&access_token=_tbroadme_
  • METHOD : GET

access token

특수한 경우(최초 계정생성, 계정 찾기 등)를 제외하고 리소스에 접근하기 위해서는 access_token 이 필요하다
access_token은 인증된 사용자에게 발급되는 짧은 키 값으로 windmill 서버에서는 access_token 유무가 로그온 여부와 동급으로 취급된다.

access_token을 입수하기 위해서는 기기인증과 사용자인증이 필요하다.

  • 기기인증 : POST-auth/authenticate(개통되는 STB), POST-auth/authenticate/device(기타장지(iPad 등))
  • 사용자인증 : POST-auth/access_token

user profile 서비스와 같이 user identity가 필요한 상황이 아닐 경우 개발자용으로 마스터 토큰을 사용할 수도 있다.
현재 마스터 토큰은 '_tbroadme_' 이다.

액세스 토큰을 얻고난 후 사용하는 방법은 두 가지가 있다.

  • Request Parameter에 넣는 방법
    Request Parameter에 access_token 을 추가한다. 
    GET me/connections?access_token=xxxxxxxxxxxxxxx
  • HTTP Header에 넣는 방법
    Request의 Authorization 헤더에 인증 정보를 추가 한다. 
    GET me/connections

Authorization: Bearer xxxxxxxxxxxxxxx

OAuth 인증 방식 활용 (Bearer token)
UPMS에서는 R2에서 HTTP Header 방식을 지원할 예정임

목록 페이지 처리

목록을 반환하는 API(예: 공지사항 목록 가져오기)들은 limit(페이지 당 item 개수), offset(아이템 번호, 0 부터 시작) 파라메터를 받는다.

Name Description Type
offset    
limit    

예를 들어 한 페이지에 10 개의 아이템이 포함된다면, 첫 페이지(offset=0&limit=10), 두 번째 페이지(offset=10&limit=10), ...

추가적으로 count(페이지 당 item 개수), page(페이지 번호, 0 부터 시작) 파라메터도 사용 가능하다.
위의 예를 다시 들자면, 첫 페이지(page=0&count=10), 두 번째 페이지(page=1&count=10), ...

목록을 시간 순서로 가져오려 할 경우 since, until 파라메터를 사용한다.
since, until은 "the milliseconds since January 1, 1970, 00:00:00 GMT" (JAVA Date 클래스와 동일)의 Long 타입(64bit) 이다.
예를 들면, 1970년 0시 0분이후의 아이템 한개를 조회할 때 (since=0&limit=1) 를 사용할 수 있다.

start_time/end_time과 같이 구간별 요청일 경우에는 since 값이 포함된 시간 부터 until 값 이전 까지의 아이템이 반환 된다. (예: 프로그램 편성)

Multilingual Documents

언어 로케일 설정에 따라 다른 값을 가질 수 있는 문서의 경우 request parameter {{lang}}으로 언어설정을 지정함으로써 해당 언어 정보만을 가져올 수 있다.
이를 생략할 경우 모든 언어에 대한 결과를 포함한다.

  • lang: GET method API 결과내의 multilingual 문서에서 원하는 언어정보만 얻을 때 사용한다 (생략했을 경우 모든 결과를 포함한다)

Application Version 정보 전달

클라이언트의 버전정보를 전달해야 하는 경우 (HTTP의 User-Agent와 유사) 서비스 API의 헤더부분에 다음과 같이 포함해서 전달해야 한다.

  • X-App-Version 헤더 : 어플리케이션 버전 정보에 대한 값은 어플리케이션이 정하는 포맷으로 실어준다. 다만, HTTP 기본 헤더 값의 문법구조는 따른다.
  • X-App-Version : 1 * ( token / version )
  • 예제 : Windmill/1.0 Tbroad/12341 xxx/1
일부 API들에서는 전달을 해도 무시하는 경우가 있을 수 있음.

Application Presence 정보 전달

사용자가 미디어의 소비 상황을 리포팅하기 위해서 다음과 같은 형태의 전달 방식을 사용하도록 한다. 이 또한 서비스 API의 헤더부분에 포함하는 형태를 이용한다.

  • X-App-Presence 헤더
  • X-App-Presence : [WINDMILL: element ] *( "," [WINDMILL: element ] )
    • element = name [ "=" [WINDMILL: value ] ] *( ";" WINDMILL:param )
    • param = name [ "=" [WINDMILL: value ] ]
  • 헤더의 길이는 1KB를 넘지않도록 한다. (API 호출 성능에 영향)
  • 예제 : type=channel; id=102;c=5;lt=123123123123;id=105;c=7;lt=21313213212
    • UseCase#1 (채널시청)
      • STB에서 채널 전환을 감지하는 어플이 사용자의 채널전환 이력을 메모리에 캐쉬하여 카운트해 간다.
      • 서비스 API를 부를때마다 메모리에 캐쉬되어 있는 정보를 헤더에 실어 보내고 보낸 데이타는 메모리에서 제거한다.
      • 채널아이디(소스아이디), N초이상 머문 카운트 (N초이상 머물러야 시청한 것으로 하는 것이 일반적임), 최근 전환시간 등을 위와 같은 규격으로 보낸다.
      • 위 예제와 같이 한번의 호출에 쌓여있는 여러 채널 정보를 flush해도 된다.
일부 API들에서는 전달을 해도 무시하는 경우가 있을 수 있으므로, 주로 사용하는 API에 대해서 사이트마다 협의해서 정하도록 한다.
이렇게 하면, 확률적으로 샘플링된 실시간 데이타를 추출해서 필요한 용도에 활용할 수 있기도 하고 (최근 전환 시간을 가지고 실시간 정보 샘플링할 수 있음)
전체 사용자에 준하는 의미있는 데이타를 수집할 수 있으면서, 서버에 특별한 추가 부하를 가하지 않는 일거양득의 방식으로 볼 수 있다.
어플의 메모리 캐쉬방식에서 오버플로우가 발생하는 경우에 대한 처리를 감안한다. FIFO방식이나 우선순위에 따른 손실처리는 가정하며 이는 사용자의 수집 행위 정보가 100퍼 전달되지 않아도 된다. (확률적인 통계는 가능)

API 문서

UPMS 서비스

현재 윈드밀 UPMS서비스에는 인증 및 계정관리와 사용자프로파일, 과금 서비스가 포함되어 있다.

인증(UPMS-auth)

윈드밀 서비스를 정상적으로 사용하기 위해서는 인증의 최종과정에서 받게되는 access_token이 필요하다.
인증과정은 access_token을 얻기 위한 과정이라 볼 수 있다.

신규 사용자가 윈드밀 서비스를 사용하기 위한 시작점은 윈드밀 서비스 외부에서 진행되는 Tbroad와 사용자의 계약에 의해 이루어지고,
이 계약에 의해 윈드밀 서버로 사용자 데이터가 provisioning 된다.

Tbroad 윈드밀 서비스는 사용자 계약 1 건당 familty id 1 개, admin id 1 개가 생성되며 기준이 되는 STB도 같이 provisioning 된다.

Provisioning이 정상적으로 되었다는 가정하에 최초 인증이 시작된다.

인증은 기기 인증과 사용자 인증, 두 가지가 있다.

기기 인증 (기준 STB)

기기 인증은 해당 기기가 윈드밀 서버에 등록된 기기임을 확인함과 동시에, 이 기기로 가입된 가입자 id를 얻기 위해 사용된다.
정상적으로 provisioning되었다면 해당기기의 unique device id (device_udid, Tbroad에서는 콜론(':')으로 구분된 소문자 MAC address 사용)가
서버 가입자 데이터베이스에 등록이 되어 있음과 동시에 해당 기기를 사용하는 family id가 등록이 되어 있다.
기기 인증은 device_udid를 기준으로 서버에 요청하여 진행하며 결과 값으로 성공 시에 가입자 id를 받게 된다.

현재 기기 인증의 부가 기능으로 PIN code 설정 기능이 있다. 이는 family_id 및 admin_id의 PIN code를 설정하는 기능으로 PIN code를 잃어버렸을
경우 초기화 기능으로 사용할 수 있다.

사용자 인증

사용자 인증과정은 정상적으로 등록된 windmill 사용자임을 확인하고 사용자 ID와 PIN code, device를 기준으로 access_token을 발급하는 과정이다.

최초 인증을 위한 사용자 ID는 기기인증을 통해 얻어진 family ID를 사용하고 기기 인증 시 초기화 시킨 PIN code를 사용하면 된다.
device id 역시 기기인증 시에 얻을 수 있다.

이렇게 인증이 끝나면 valid access_token을 받게 된다.
access_token은 해당 기기에 해당 유저별로 생성이 된다.
만약 사용자 인증을 여러 번 요청을 하게 되면 특별한 이유가 없는 한 기존에 발급된 access_token을 재활용한다. (This is subject to change)

각종 키

인증 과정에서 사용되는 키에 대한 설명이다.

  • Application Key : 클라이언트 종류별로 발급되는 키로 현재는 Tbroad용 STB windmill client로 "windmillClient1" 라는 값과 handheld client용으로 "windmillClient2" 라는 값을 사용한다.
  • Secrete Key : 인증 과정에서 hash 값을 만들기 위해 사용하는 키로 현재는 "windmillUserprofile1" 라는 값을 사용한다.

계정관리 (UPMS-accounts)

윈드밀 계정 체계

Tbroad의 윈드밀 서비스에서는 family 계정, admin 계정, 일반 계정 등의 계정 종류가 있다.

family계정은 윈드밀 시스템으로 Provisioning Server를 통해 Provisioning 요쳥이 올 경우 생성이 된다.
따라서, 현재 family 계정은 계약과 1:1 관계(1건의 가입계약 = 1 family id)를 가지고 있다.
다만, 계약과 1:1관계라고 해서 법인/자연인과 1:1관계가 되는 상황은 아닐 수 있다. (Tbroad 계약 처리 방식에 따라 달라짐)

family 계정은 주 billing 계정이며 계정관리 API로는 생성/삭제가 불가능하다.

admin 계정은 family 계정이 생성될 때 쌍으로 생성이 되며 이 역시 계정 관리 API로는 생성/삭제가 불가능하다.
특정 API를 사용하기 위해서는 admin 계정의 access_token이 필요하다.

일반 계정은 Provisioning 시에 생성되지 않으며 windmill 계정관리 API를 통해 생성해야지만 생성이 된다.
보조 기기 (태블릿, 스마트폰 등의 companion device)용으로 계정을 추가할 수 있다.

사용자 장치(UPMS-user-device)

청약을 통해 이미 등록된 기기들을 제외하고 나머지 기기들(스마트폰, 태블릿 등)이 윈드밀 서비스를 사용하기 위해서는 기기들을 등록하고
인증을 받아야 한다. 사용자 장치 API를 활용하여 기기들을 등록할 수 있다.

Pairing

등록되지 않은 기기들을 등록하여 현재 사용자와 연결하는 과정이 pairing 과정이다.

Pairing과정은 다음과 같은 틀에서 진행된다.

  1. STB에서 윈드밀 서버로 OTP 생성 요청 (UPMS-users-id-devices-otp)
  2. TV 화면에 OTP를 보여주고 pairing하기 위한 기기(컴패니언 기기)에서 OTP 입력 요청
  3. 컴패니언 기기에서 입력된 OTP를 가지고 기기 인증 요청 (UPMS-auth-authenicate-device)
  4. 서버에서 OTP를 기준으로 valid OTP 입력 시 기기를 등록하고 가입자 ID 및 리스트 전송
  5. 가입자 리스트에서 자신의 id를 찾아 access_token요청 (UPMS-auth-access_token)

상기 과정이 끝나면 컴패니언 기기에서도 유효한 access_token을 가지고 윈드밀 서비스를 사용할 수 있다.

메뉴는 STB에서 보여지는 메뉴를 구성한 것이다.

윈드밀에서는 메뉴 구조를 서비스 데이터화 하여 별도의 어플리케이션 송출 없이 메뉴 구조를 변경할 수 있도록 되어 있다.
